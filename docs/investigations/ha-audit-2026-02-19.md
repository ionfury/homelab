## CRITICAL Findings (Immediate Risk)

### 1. Longhorn Backup Target Not Configured — DATA LOSS RISK

- BackupTarget URL is **empty** despite Helm values specifying `s3://homelab-longhorn-backup-${cluster_name}@us-east-2/`
- All 6 recurring backup jobs (daily + frequent) **silently fail** — zero backup volumes exist
- The 13 volumes with 3 Longhorn replicas survive a single node failure, but a cluster-wide event (power, fire, simultaneous disk failures) means **total data loss** for: 10TiB media library, 500GB immich photos, game saves, monitoring history
- **Likely cause:** `${cluster_name}` substitution failure or Longhorn v1.11.0 not propagating `defaultSettings.backupTarget` to the BackupTarget CRD

### 2. CNPG WAL Archiving Failing — DATABASE BACKUP RISK

- Both CNPG ScheduledBackups show `phase: walArchivingFailing`
- Paradoxically, cluster status reports `ContinuousArchiving=True` — conflicting signals
- If WAL archiving is truly broken, **no point-in-time recovery** is possible for either the platform or immich databases
- 3-instance replication protects against node failure, but not against logical corruption or cluster-wide events

### 3. CoreDNS — Both Replicas on node2

- 2 replicas, but both scheduled on node2 due to soft anti-affinity losing to resource scoring
- **If node2 fails: total DNS blackout cluster-wide** — no pod can resolve any service name or external domain
- This is the single most impactful HA failure: DNS is a dependency of everything

### 4. Authentication Chain — All Single Replicas

- **Authelia** (node42), **LLDAP** (node41), **oauth2-proxy** (node2) — each is 1 replica
- Any one of these failing breaks **all authenticated access** to internal services
- These are in the critical path for every HTTPRoute using `platform-auth-policy`

## HIGH Findings (Significant Risk)

### 5. node2 Concentration — Systemic Risk

If node2 goes down, you simultaneously lose:

| Layer | Services Lost |
|-------|--------------|
| **DNS** | Both CoreDNS pods |
| **GitOps** | 6 of 7 Flux controllers (source, kustomize, helm, operator, image-*) |
| **Certificates** | cert-manager + webhook + cainjector |
| **Monitoring** | Prometheus + Loki + Grafana + kube-state-metrics |
| **Secrets** | external-secrets + webhook |
| **Mesh** | istiod control plane |
| **Health** | canary-checker (blocks promotion pipeline) |

### 6. Istiod — Single Replica

- 1 instance on node2, HPA configured but min=1
- Loss means no xDS config updates, no new sidecar injection, no mesh policy changes
- Existing connections survive (ztunnel caches config), but new connections and config changes fail

### 7. Prometheus — Single Replica

- 1 instance on node2, no Thanos sidecar, no remote write
- Loss means **total alerting blackout** — Alertmanager has 3 replicas but nothing to feed it alerts
- 14-day retention, 260K active time series, 0.96 GiB TSDB

### 8. Loki — SingleBinary Mode, 1 Replica

- Single instance on node2, replication_factor=1
- Loss means all log ingestion stops; Alloy DaemonSets buffer locally but eventually drop logs

### 9. No PDBs on Gateway Deployments

- external-istio (3 replicas) and internal-istio (3 replicas) have **no PodDisruptionBudgets**
- During Tuppr-managed node upgrades, voluntary drain could evict all replicas simultaneously

## What's Working Well

- **Control plane**: 3-node etcd cluster, correct quorum sizing
- **Alertmanager**: 3 replicas, properly meshed, spread across 3 nodes
- **Dragonfly**: 3 replicas, enforced topology spread (`DoNotSchedule`), PDB, S3 snapshots
- **CNPG databases**: 3-instance clusters with streaming replication, PgBouncer poolers, PDBs
- **DaemonSets**: Cilium, Alloy, node-exporter, ztunnel, istio-cni, spegel — all 4/4
- **Longhorn volume design**: Intentional 1-replica for app-replicated data (CNPG, Dragonfly, Garage), 3-replica for everything else
- **Dual L2+BGP load balancing**: Defense in depth with fast failover timers
- **StorageClass taxonomy**: Well-designed ephemeral/fast/slow/fast-nr classes matching use cases
- **Istio ambient mode**: ztunnel DaemonSet provides per-node data plane coverage
